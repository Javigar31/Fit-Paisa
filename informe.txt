**Fecha:** 2026-04-09
**Estado:** Estable en Producción (`master`) y Testing (`testing`).

## 1. Cambios en la Arquitectura (Cache-Busting)
*   **Renombramiento Crítico**: Se ha migrado `fitpaisa_cliente.html` a `fitpaisa_cliente-v2.html` para invalidar cachés agresivas en redes corporativas (Fortinet).
*   **Redirecciones**: Actualizadas todas las referencias en `index.html` (login y registro) para apuntar a la nueva versión `-v2`.

## 2. Sistema de Mensajería (Dinamización)
*   **Backend**: `api/messages.php` operativo con acciones `inbox`, `thread` y `send`.
*   **Frontend**: Se eliminaron todos los datos mock (Laura Vidal). 
    *   Implementadas `loadLastMessage()` y `fetchInbox()` para poblar la UI desde la base de datos.

*   **Bug de JS**: Corregido un `TypeError` en `saveProfile` donde se intentaba acceder a un objeto intermedio `.data` que ya no existía en el nuevo formato de respuesta de la API.

*   **Lógica de Macros**: La función `calculate_macros` ahora aplica correctamente la diferencia de peso multiplicada por 7700 kcal repartida entre el tiempo objetivo.
*   **UI Dinámica**: El TMB y el ajuste calórico (Déficit/Superávit) en `fitpaisa_cliente-v2.html` ya no son texto estático; se actualizan mediante IDs `diet-tmb-val` y `diet-adj-val`.

## 5. Acciones Manuales Pendientes
*   **Migración de DB**: Es necesario ejecutar `api/db_migration.php` en cualquier entorno nuevo o base de datos de producción para añadir las columnas de objetivos a la tabla `profiles`.
*   **Roles**: Las credenciales de entrenador registradas hoy son `javigar.1731@hotmail.com` / `Javigar.coach1731` (Rol: `COACH`).

# Informe Técnico (Tarde): Resolución Definitiva de Producción (v1.3.2)
**Fecha:** 2026-04-10
**Estado:** Producción (`master`) 100% operativa y estabilizada.

## 1. Resolución del Error 500 (Login en Producción)
*   **Problema Detectado**: Fallo de autenticación en la base de datos (`password authentication failed`).
*   **Causa Raíz**: Vercel inyecta automáticamente variables de entorno (`POSTGRES_PASSWORD`) que en este proyecto están obsoletas o bloqueadas y no se pueden modificar desde el panel. El código las priorizaba, causando el fallo.
*   **Solución Técnica** (`api/_db.php`):
    *   **Prioridad de Credenciales**: Se ha modificado la lógica para que el sistema use primero la variable manual `DB_PASSWORD_NUEVA` antes que cualquier variable automática de Vercel.
    *   **Forzado de Base de Datos**: 
        *   La rama `master` está forzada por código a usar la base de datos `neondb`.
        *   La rama `testing` está forzada por código a usar `fitpaisa_testing`.

## 2. Herramientas de Diagnóstico
*   **Script Avanzado**: Se desarrolló `api/diag_v3.php` para realizar pruebas de aislamiento (PDO puro) y verificar la visibilidad de variables de entorno sin exponer secretos. 
*   **Seguridad**: Tras la estabilización, todos los archivos `diag_*.php` han sido eliminados de las ramas para evitar vulnerabilidades de exposición de información técnica.

## 3. Lecciones Aprendidas para el Futuro
*   **Variables de Vercel**: No confiar en las variables inyectadas automáticamente por las integraciones si se han realizado cambios manuales en Neon. Siempre priorizar variables de usuario (`_PROD` o manuales).
*   **Aislamiento de Errores**: Ante un Error 500 genérico de PHP, usar siempre un script que pruebe la conexión PDO pura fuera del framework/wrapper de la app para ver el mensaje de error real de Postgres.

## 4. Estado Final
*   **Master**: Conectado a `neondb` - Operativa.
*   **Testing**: Conectado a `fitpaisa_testing` - Operativa.

# Informe Técnico (Optimización): UI y Responsividad (v1.3.3)
**Fecha:** 2026-04-10
**Estado:** Interfaz de cliente optimizada y 100% responsiva.

## 1. Mejoras en la Arquitectura Responsiva
*   **Problema**: El diseño fijo de `100vh` en escritorio recortaba contenido en pantallas de poca altura o resoluciones bajas, perdiendo visibilidad de los macros y el calendario.
*   **Solución (Centrado Seguro)**: Se implementó una lógica de márgenes dinámicos (`margin: auto 0`) en los paneles internos (`.slide-hero`, `.slide-panel`).
    *   **Resultado**: El contenido se mantiene centrado si hay espacio, pero si la ventana es pequeña, el el scroll permite ver todo el contenido empezando desde el inicio (sin recortes superiores).
*   **Estética Premium**: Se ocultaron las barras de scroll mediante CSS para mantener la sensación de aplicación nativa, manteniendo la funcionalidad de desplazamiento intacta.

## 2. Correcciones en Componentes Críticos
*   **Catálogo de Alimentos**: Se corrigió el buscador cambiando las sugerencias a `position: absolute`. Esto evita que el desplegable "empuje" el formulario hacia abajo, permitiendo una experiencia de búsqueda fluida.
*   **Calendario Nutricional**: Se eliminó el `justify-content: space-between` para permitir un agrupamiento natural de los días y se ajustó la altura mínima para evitar saltos visuales durante la carga.

## 3. Estado del Repositorio
*   **Cambios Sincronizados**: Todas las mejoras visuales han sido fusionadas y subidas a `master` tras ser validadas en la rama `testing`.
*   **Próximos Pasos**: Continuar con la mejora de funcionalidades backend y paneles de administración bajo esta nueva base responsiva.

# Informe Técnico (Crítico): Restauración de Master (v1.5.2)
**Fecha:** 2026-04-11
**Estado:** Producción estabilizada. Error 500 resuelto.

## 1. Resolución del Error 500 en Registro
*   **Problema**: El registro de alimentos fallaba sistemáticamente en producción con un Error 500.
*   **Causa Raíz**: El script de auto-migración omitía las columnas `portion_amount` y `portion_unit` en la tabla `food_entries` de la base de datos `neondb`. 
*   **Solución**: Se completó la función `fp_ensure_schema()` en `api/_db.php`. 
    *   Al abrir el diagnóstico o registrar un alimento, el sistema autodetecta las columnas faltantes y las crea en tiempo real sin intervención manual.
*   **Resultado**: Registro de alimentos 100% operativo en `https://fit-paisa.vercel.app/`.

# Informe Técnico (Tarde): Resolución Definitiva de Producción (v1.3.2)
**Fecha:** 2026-04-10
**Estado:** Producción (`master`) 100% operativa y estabilizada.

## 1. Resolución del Error 500 (Login en Producción)
*   **Problema Detectado**: Fallo de autenticación en la base de datos (`password authentication failed`).
*   **Causa Raíz**: Vercel inyecta automáticamente variables de entorno (`POSTGRES_PASSWORD`) que en este proyecto están obsoletas o bloqueadas y no se pueden modificar desde el panel. El código las priorizaba, causando el fallo.
*   **Solución Técnica** (`api/_db.php`):
    *   **Prioridad de Credenciales**: Se ha modificado la lógica para que el sistema use primero la variable manual `DB_PASSWORD_NUEVA` antes que cualquier variable automática de Vercel.
    *   **Forzado de Base de Datos**: 
        *   La rama `master` está forzada por código a usar la base de datos `neondb`.
        *   La rama `testing` está forzada por código a usar `fitpaisa_testing`.

## 2. Herramientas de Diagnóstico
*   **Script Avanzado**: Se desarrolló `api/diag_v3.php` para realizar pruebas de aislamiento (PDO puro) y verificar la visibilidad de variables de entorno sin exponer secretos. 
*   **Seguridad**: Tras la estabilización, todos los archivos `diag_*.php` han sido eliminados de las ramas para evitar vulnerabilidades de exposición de información técnica.

## 3. Lecciones Aprendidas para el Futuro
*   **Variables de Vercel**: No confiar en las variables inyectadas automáticamente por las integraciones si se han realizado cambios manuales en Neon. Siempre priorizar variables de usuario (`_PROD` o manuales).
*   **Aislamiento de Errores**: Ante un Error 500 genérico de PHP, usar siempre un script que pruebe la conexión PDO pura fuera del framework/wrapper de la app para ver el mensaje de error real de Postgres.

## 4. Estado Final
*   **Master**: Conectado a `neondb` - Operativa.
*   **Testing**: Conectado a `fitpaisa_testing` - Operativa.

# Informe Técnico (Incidencias): Estabilización Fallida (v2.1.x a v2.8.x)
**Fecha:** 2026-04-12
**Estado:** REVERTIDO a v2.0.0 (Commit 79defe0).

## 1. Resumen de la Jornada
Se intentó estabilizar la autenticación y los paneles de control en el entorno de Vercel (PHP 8.3), resultando en una serie de errores 500 persistentes y regresiones. Se decidió dar marcha atrás para mantener la integridad del código base.

## 2. Hallazgos Críticos (Causa Raíz)
*   **Conflicto de Credenciales**: Se identificó que Vercel inyecta una variable `PGPASSWORD` que falla en Neon. Nuestra base de datos requiere `DB_PASSWORD_NUEVA`. Cualquier script que no priorice explícitamente esta última fallará con "password authentication failed".
*   **Inestabilidad de Inclusiones**: Los archivos monolíticos (como `auth_direct.php`) funcionaron donde los controladores estándar fallaron. Esto sugiere que el sistema de `require_once` o el cargado de `_db.php` está provocando fallos por:
    1.  **Límites de Conexión**: Demasiadas conexiones abiertas al pooler simultáneamente.
    2.  **Headers Already Sent**: Posible presencia de caracteres invisibles o salida accidental antes de las cabeceras JSON.
*   **Latencia de Despliegue**: Se observó que Vercel no siempre actualiza los archivos `api/*.php` de forma instantánea, provocando que pruebas de diagnóstico mostraran resultados obsoletos (v2.2.1 vs v2.3.0).

## 3. Pendientes para el Futuro
*   **Migración Monolítica**: Si los errores 500 persisten, la solución definitiva será fusionar la lógica de `_db.php` y `_jwt.php` directamente dentro de cada controlador principal para eliminar el overhead de inclusiones.
*   **Depuración de Panel**: Existe un error de sintaxis JS en `fitpaisa_entrenador.html` (mismatched braces en `checkAuth`) que debe corregirse en la próxima sesión.

## 4. Decisión Técnica
Se realiza un `git reset --hard 79defe0` para limpiar la rama de pruebas de trazados de depuración (`try-catch` masivos) y archivos temporales (`auth_direct.php`, `check_files.php`).

01. Objetivos de Estabilidad

El foco principal es garantizar la resiliencia absoluta tras la migración a controladores monolíticos.

  • Validar la conexión a Neon DB ante latencias críticas de Vercel (PHP 8.3).
  • Asegurar un rendimiento constante de 60 FPS en el motor de partículas 3D.
  • Garantizar que el sistema JWT bloquea accesos no autorizados en los nuevos endpoints consolidados.
Foco Arquitectónico

02. Arquitectura Monolítica

Nuestra mayor defensa contra los errores 500 es la autonomía de los controladores críticos (auth, profile, nutrition, workouts).

Protocolo de Validación Atómica:

  • Test de Aislamiento: Los controladores deben operar correctamente incluso si los archivos base (_db.php, _jwt.php) son removidos.
  • Sincronía de Helpers: Auditoría de las funciones duplicadas para evitar divergencias lógicas.
  • Métricas de Latencia: Evaluación del impacto del tamaño de archivo en los Cold Starts de Vercel.

03. Tipos de Pruebas (Metodología)

Caja Blanca

Auditoría de los flujos try-catch internos y la lógica de saneamiento de datos en el backend.

Caja Negra

Validación de la integridad del JSON de salida independientemente de la estructura interna del archivo.

Regresión

Uso estricto de la rama 'testing' para asegurar que el motor 3D y la API v3.2 son estables.

04. Protocolo de Depuración

Estructuramos nuestras herramientas bajo tres dimensiones críticas:

  • Dónde: Scripts dinámicos diag_*.php como puntos de ruptura virtuales.
  • Cómo: Ejecución en aislamiento monolítico para detectar colisiones de redemarcación.
  • Qué: Examinadores de variables mediante logs controlados y respuestas JSON detalladas.

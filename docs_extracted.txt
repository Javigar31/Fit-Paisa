1. Vista de Alto Nivel 
FitPaisa se concibe como una plataforma web integral para la gestión del 
entrenamiento físico y la nutrición personalizada. Visto como caja negra, el sistema 
recibe datos del usuario (perfil físico, comidas, sesiones de entrenamiento) y devuelve 
planes adaptados, estadísticas de progreso y comunicación directa con su 
entrenador. 
El sistema opera de forma centralizada: el usuario interactúa con un entorno web 
responsive, los entrenadores supervisan el progreso de sus clientes en tiempo real y 
el administrador gestiona la plataforma desde un panel con permisos elevados. 
FitPaisa actúa como asistente activo y dinámico, integrando servicios externos para 
pagos y notificaciones.

2. Patrones Arquitectónicos 
Para FitPaisa se ha definido una arquitectura híbrida que combina el modelo Cliente-
Servidor con el patrón de diseño Modelo-Vista-Controlador (MVC). Esta decisión se 
fundamenta en las características del proyecto: equipo unipersonal, entorno 
académico, requisito de respuesta inferior a 2 segundos y acceso exclusivamente 
web.

2.3 Justificación de la decisión 
• 
Adecuación al equipo: la arquitectura MVC es manejable por un desarrollador 
unipersonal sin necesidad de orquestación de servicios. 
• 
Cumplimiento de RF12: el patrón MVC facilita flujos de registro de máximo 3 
pasos mediante controladores dedicados. 
• 
Rendimiento: la separación de capas permite optimizar cada nivel de forma 
independiente para cumplir el requisito de carga en < 2 segundos. 
• 
Escalabilidad futura: la separación Cliente-Servidor permite migrar a 
microservicios en el futuro sin reestructurar la lógica de negocio.

4.1.3 Algoritmo de Autenticación (RF1) 
El proceso de login sigue los siguientes pasos en estricto orden: 
1. Recibir email y contraseña desde el formulario del Frontend. 
2. Buscar el usuario en la BD por email. Si no existe → retornar error 401. 
3. Verificar que isActive = true. Si inactivo → retornar error 403. 
4. Comparar contraseña recibida con passwordHash usando bcrypt.compare(). 
5. Si no coincide → incrementar contador de intentos fallidos. Si ≥ 5 → bloquear 
temporalmente. 
6. Si coincide → generar token JWT con payload: {userId, email, role, exp: +2h}. 
7. Registrar timestamp de último acceso y retornar token + datos de sesión.

4.3.3 Algoritmo de Cálculo de Macronutrientes 
El motor de macronutrientes calcula los objetivos diarios usando las fórmulas estándar 
de nutrición deportiva: 
 
Paso 1 — Cálculo del Metabolismo Basal (TMB) con la ecuación de Mifflin-St Jeor: 
• 
Hombre: TMB = (10 × peso_kg) + (6.25 × talla_cm) − (5 × edad) + 5 
• 
Mujer: TMB = (10 × peso_kg) + (6.25 × talla_cm) − (5 × edad) − 161 
 
Paso 2 — Gasto Energético Total (GET) = TMB × Factor de Actividad:

4.7.3 Algoritmo de Procesamiento de Pago 
8. El Frontend redirige al usuario a la URL de checkout generada por 
createCheckout(). 
9. El usuario completa el pago en la interfaz segura de Stripe/PayPal (PCI-DSS 
compliant). 
10. La pasarela envía un webhook al endpoint /api/payments/webhook con el 
evento de pago. 
11. handleWebhook() valida la firma del evento usando la clave secreta de la 
pasarela. 
12. Si el evento es payment_intent.succeeded o 
PAYMENT.CAPTURE.COMPLETED → activateSubscription(). 
13. Si el evento indica fallo → persistir estado CANCELLED y notificar al usuario 
por email. 
14. En ambos casos → registrar el evento completo en el log de auditoría.

7. Manejo de Errores y Excepciones 
7.1 Estrategia General 
Todos los módulos implementan el patrón de manejo centralizado de errores. Las 
excepciones no controladas se capturan en el middleware global del controlador, que

1. Introducción 
1.1 Propósito 
El presente documento describe el Diseño de Alto Nivel (High Level Design — HLD) 
del sistema FitPaisa, una plataforma web integral para la gestión del entrenamiento 
físico y la nutrición personalizada. Define la estructura modular del sistema, los 
subsistemas principales, sus responsabilidades funcionales y las interfaces de 
comunicación entre ellos, transformando los requisitos de la ERS en una organización 
técnica implementable. 
Este documento está dirigido al equipo de desarrollo, al arquitecto del sistema y a los 
evaluadores académicos del proyecto. 
 
1.2 Alcance del Sistema 
FitPaisa es una aplicación web independiente accesible desde navegadores 
modernos que proporcionará las siguientes capacidades: 
• 
Registro y autenticación segura de usuarios con tres roles: Usuario, 
Entrenador y Administrador. 
• 
Gestión de perfil físico: peso, altura, edad, objetivos y nivel de actividad. 
• 
Planes personalizados de entrenamiento y nutrición, asignados y 
supervisados por entrenadores. 
• 
Registro diario de comidas con cálculo automático de macronutrientes.

• 
Registro de entrenamientos: ejercicios, series, repeticiones y cargas 
máximas. 
• 
Estadísticas y gráficos de evolución física y nutricional. 
• 
Mensajería interna entre usuarios y entrenadores. 
• 
Suscripciones Premium mediante pasarelas de pago externas (Stripe / 
PayPal). 
• 
Notificaciones y recordatorios automáticos por correo electrónico. 
El sistema no sustituye asesoramiento médico ni realiza diagnósticos clínicos.

5. Flujo General de Operación 
El flujo estándar de operación para los dos casos de uso principales del sistema es el 
siguiente: 
 
5.1 Registro de comida y cálculo de macros 
1. El usuario carga la aplicación web en el navegador (Frontend). 
2. Se verifica la sesión activa mediante el Módulo de Autenticación (token JWT). 
3. El usuario accede al formulario de registro de comida en el dashboard. 
4. El usuario introduce el alimento y la cantidad en gramos. 
5. El Controlador recibe el dato y el Módulo de Nutrición calcula los 
macronutrientes automáticamente. 
6. El resultado se persiste en la base de datos (FOOD_ENTRIES). 
7. El Frontend actualiza el gráfico de progreso nutricional en menos de 2 
segundos. 
8. Si se supera el objetivo calórico diario, el Servicio de Notificaciones envía una 
alerta por email. 
 
5.2 Creación y aprobación de plan de entrenamiento 
9. El entrenador accede a su panel y crea un nuevo plan de entrenamiento 
(estado DRAFT). 
10. Añade ejercicios, series, repeticiones y días de la semana. 
11. Envía el plan para revisión del usuario (estado PENDING_APPROVAL). 
12. El usuario revisa el plan propuesto en su dashboard (RF3). 
13. El entrenador aprueba el plan definitivo (estado ACTIVE). 
14. El Servicio de Notificaciones envía un email al usuario confirmando la 
activación del plan (RF9). 
15. El usuario comienza a registrar sesiones de entrenamiento contra el plan 
activo (RF5).

1. Introducción 
1.1 Propósito 
El propósito de este documento es describir de forma clara, detallada y 
estructurada los requisitos del sistema FitPaisa, siguiendo el estándar IEEE 830. 
Este documento sirve como referencia principal para el desarrollo, validación y 
evaluación del sistema, y está dirigido a desarrolladores, responsables del 
proyecto, docentes y futuros mantenedores. 
1.2 Ámbito del Sistema 
FitPaisa es una plataforma web orientada a la gestión integral del entrenamiento 
físico y la nutrición personalizada. El sistema permitirá a los usuarios finales 
registrar su información física, objetivos personales y hábitos alimenticios, con el 
fin de recibir planes adaptados a sus necesidades. 
El sistema proporcionará herramientas para el seguimiento diario de 
entrenamientos y comidas, permitiendo visualizar la evolución mediante 
estadísticas y gráficos. Además, facilitará la comunicación directa entre usuarios y 
entrenadores, mejorando el acompañamiento y la personalización del proceso. 
El sistema no sustituye asesoramiento médico ni realiza diagnósticos clínicos, 
limitándose al ámbito del entrenamiento físico y la nutrición general. 
1.3 Definiciones, Acrónimos y Abreviaturas 
• ERS: Especificación de Requisitos Software. 
• RF: Requisito Funcional. 
 
 
 
 
 
• NFR: Requisito No Funcional. 
• Usuario: Persona registrada que utiliza la plataforma para entrenar y 
registrar su progreso. 
• Entrenador: Profesional que gestiona planes y seguimiento de usuarios. 
• Administrador: Usuario con permisos avanzados de gestión del sistema. 
1.4 Referencias 
• IEEE Std. 830-1998 – Recommended Practice for Software Requirements 
Specifications. 
• Documento de alcance del proyecto FitPaisa. 
1.5 Visión General del Documento 
Este documento se divide en cuatro secciones principales: una introducción 
general, una descripción del contexto del sistema, la especificación detallada de 
los requisitos y un apartado de apéndices con información complementaria. 
 
2. Descripción General 
2.1 Perspectiva del Producto 
FitPaisa es una aplicación web independiente que funcionará a través de 
navegadores modernos. El sistema se estructura en distintos roles de usuario, 
cada uno con funcionalidades específicas, y se apoya en servicios externos como 
pasarelas de pago y sistemas de notificaciones. 
2.2 Funciones del Producto 
Las funciones principales del sistema incluyen la gestión de cuentas de usuario, la 
creación y asignación de planes personalizados de entrenamiento y nutrición, el 
registro diario de actividades y comidas, la visualización del progreso físico y la 
comunicación interna entre usuarios y entrenadores. 
 
 
 
 
 
2.3 Características de los Usuarios 
Los usuarios finales tendrán conocimientos básicos en el uso de aplicaciones web. 
Los entrenadores contarán con experiencia en entrenamiento físico y/o nutrición, 
mientras que los administradores dispondrán de conocimientos técnicos medios 
para la gestión del sistema. 
2.4 Restricciones 
El sistema deberá ser accesible desde navegadores web compatibles. El uso de 
pasarelas de pago externas condiciona ciertos procesos del sistema. Además, se 
deberán cumplir normativas básicas de seguridad y protección de datos. 
2.5 Suposiciones y Dependencias 
Se asume que los usuarios disponen de conexión a internet y utilizan navegadores 
actualizados. El correcto funcionamiento del sistema depende de servicios 
externos como correo electrónico y pagos. 
2.6 Requisitos Futuros 
Como posible mejora futura se contempla únicamente la ampliación de 
funcionalidades dentro del entorno web. 
 
3. Requisitos Específicos 
3.1 Interfaces Externas 
El sistema dispondrá de una interfaz web accesible desde navegadores modernos, 
diseñada bajo criterios de usabilidad y diseño responsive. Asimismo, se integrará 
con servicios externos como pasarelas de pago (Stripe y/o PayPal) y servicios de 
envío de correos electrónicos para notificaciones y recordatorios. 
3.2 Requisitos Funcionales 
 
 
 
 
 
• RF1: El sistema permitirá a los usuarios registrarse mediante correo electrónico 
y contraseña, así como autenticarse de forma segura. 
 
• RF2: El usuario podrá completar y modificar su perfil físico, incluyendo datos 
como peso, altura, edad, objetivos y hábitos generales. 
 
• RF3: El sistema permitirá al usuario consultar en cualquier momento su plan 
personalizado de entrenamiento y nutrición. 
 
• RF4: El usuario podrá registrar diariamente sus comidas, y el sistema calculará 
los macronutrientes correspondientes. 
 
• RF5: El usuario podrá registrar sus entrenamientos indicando ejercicios, series, 
repeticiones y cargas utilizadas. 
 
• RF6: El entrenador podrá consultar, editar y aprobar los planes asignados a sus 
clientes. 
 
• RF7: El sistema permitirá la comunicación directa entre usuarios y entrenadores 
mediante mensajería interna. 
 
• RF8: El usuario podrá subir fotografías de progreso y visualizar estadísticas de 
evolución física. 
 
• RF9: El sistema enviará notificaciones y recordatorios relacionados con 
entrenamientos, comidas y actividades pendientes. 
 
• RF10: El administrador podrá gestionar usuarios, entrenadores, planes y 
permisos del sistema. 
 
 
 
 
 
 
• RF11: El usuario podrá suscribirse a planes de pago mediante pasarelas 
externas. 
 
• RF12: El proceso de registro y primer acceso no deberá superar los tres pasos. 
3.3 Requisitos de Rendimiento 
El sistema deberá responder a las solicitudes del usuario en un tiempo inferior a 
dos segundos en la carga del panel principal. Asimismo, deberá soportar un 
mínimo de un número reducido de usuarios activos, acorde a un proyecto 
académico simultáneos sin degradación apreciable del servicio. 
3.4 Restricciones de Diseño 
El diseño del sistema estará basado en una arquitectura web estándar. Se deberá 
garantizar la compatibilidad con los navegadores Chrome, Firefox y Edge en sus 
versiones actuales. 
3.5 Atributos del Sistema 
El sistema deberá garantizar altos niveles de seguridad, disponibilidad y privacidad. 
Toda la información se transmitirá mediante protocolos cifrados (TLS 1.2 o 
superior) y los datos sensibles se almacenarán de forma segura. La disponibilidad 
mínima esperada será del 99,5% mensual.
